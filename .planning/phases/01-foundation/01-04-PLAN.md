<task type="checkpoint:human-action">
  <name>Configurar Resend SMTP no painel do Supabase + verificar DNS (EMAIL-01, EMAIL-02)</name>
  <what-built>No application code changes. This is an operator panel configuration task. The DEPLOY-CHECKLIST.md (created in Plan 05) has the reference documentation. This task applies the configuration.</what-built>
  <how-to-verify>
**STEP 1 — Verify domain in Resend:**
1. Log in to [resend.com](https://resend.com) → **Domains** → **Add Domain**
2. Enter the sending domain (e.g., `mdhe.com.br` or the configured sending domain)
3. Resend shows DNS records to add. Add ALL of:
   - **SPF**: TXT record — `v=spf1 include:amazonses.com ~all` (or the exact value Resend shows)
   - **DKIM**: TXT record on `resend._domainkey.yourdomain.com` with the long key Resend provides
   - **DMARC** (recommended): TXT record on `_dmarc.yourdomain.com`: `v=DMARC1; p=none; rua=mailto:dmarc@yourdomain.com`
4. Add records at your DNS provider (e.g., Cloudflare DNS → TXT records)
5. Wait for propagation (5–30 minutes typical). Click **Verify** in Resend — all records should show green.

**STEP 2 — Create Resend API key:**
1. resend.com → **API Keys** → **Create API Key**
2. Name: `supabase-smtp`
3. Permission: **Sending access** (NOT Full Access)
4. Save the key — it is shown only once. Store it in your password manager.

**STEP 3 — Configure custom SMTP in Supabase:**
1. Supabase Dashboard → **Project Settings** → **Auth** → scroll to **SMTP Settings**
2. Toggle **Enable Custom SMTP** to **ON**
3. Fill in:
   - **Host:** `smtp.resend.com`
   - **Port:** `465`
   - **User:** `resend`
   - **Password:** the API key from Step 2
   - **Sender Email:** your verified domain address (e.g., `notificacoes@mdhe.com.br`)
   - **Sender Name:** `MDHE — Gestão de Incidentes`
4. Click **Save**

**STEP 4 — Verify deliverability (EMAIL-02):**
1. In Supabase Auth → trigger a **"Forgot password"** flow using a Gmail test address
   - Confirm: email arrives in Gmail inbox (not spam) within 60 seconds
   - Confirm: sender shows your `@mdhe.com.br` domain
2. Trigger a **"User invite"** (or another password-reset) using an Outlook/Hotmail address
   - Confirm: email arrives in Outlook inbox (not spam) within 60 seconds
   - If either goes to spam: re-check SPF/DKIM in Resend dashboard (all three records should be green)

**STEP 5 — Document in DEPLOY-CHECKLIST.md:**
Open `docs/DEPLOY-CHECKLIST.md` and fill in the placeholder fields in the email section:
- The `EMAIL_FROM` address you configured (e.g., `notificacoes@mdhe.com.br`)
- The Resend account login (e.g., your email at resend.com)
- Where the API key is stored (e.g., "1Password vault: MDHE Production")
- The DNS provider where SPF/DKIM records were added
  </how-to-verify>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Supabase Auth → Resend SMTP | Email credentials (API key) transmitted only in the Supabase Auth panel SMTP password field — never in application code |
| Resend → recipient inbox | SPF/DKIM records establish domain authenticity; DMARC provides monitoring |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T1 | Information Disclosure | Resend API key | mitigate | API key is entered only in Supabase Auth SMTP panel (server-side config), never in application code, never in `src/lib/env.ts`, never committed to git. Create with "Sending access only" permission (principle of least privilege). |
</threat_model>
